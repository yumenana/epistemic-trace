**一个为 LLM 协同研究设计的 Append-Only 推理日志。**

## 问题

你在一个研究项目里干了六个月, AI 助手帮你写了大部分代码。

今天你问它: *"这里加个 global residual connection 怎么样?"*

它愉快地写了 patch。你盯着 diff 看了十秒钟才想起来 — 第三周你就试过了。这是条死路。你花了两天找出原因。**大费周章可能最后又走进这个死胡同**

你也许会反驳: README 不就是干这个的吗? 注释呢? — 但 README 的主要内容是结果, 顶多附带一句"为什么是这个结论"; 注释贴着代码, 重构一来就没了, 留下的最多是一句"为什么重构"。它们都不会记得你当初 *为什么* 走这条路而不是另一条, 你最先试了什么, 失败了什么, 又从那次失败中学到了什么。git log 也只是"什么时候改的什么", 不是"为什么没选另一种改法"。

而 AI 助手, 每开一个新对话都从零开始。同样的约束你解释一遍, 同样的死路你警告一遍, 同样靠血泪换来的直觉你重述一遍。有时候你来不及制止, 它就自信满满地把你早就废弃的方案重新搭了一遍。

**Epistemic Trace 是一个为人和 LLM 共同阅读而设计的结构化推理日志。** 它跨会话保留思维链, 把"失败"当作重要信息, 并强制每个假说声明"什么证据能推翻我"。

## 它长什么样

一份 Trace 由两个区组成: 顶部是 **工作假说**, 下方是按时间排列的 **认知事件时间线**。

```markdown
### H4: 主干重建流中避免归一化
> #normalization #amplitude-phase
> ↗ 适用范围被 H10 [2026-04-19] 修正 — decoder 幅值累积节点需 GroupNorm

不是"超分不需要归一化"这种笼统结论。具体原因:

1. [推测] 归一化强迫零均值+单位方差。动漫 4x 超分中, 平坦色块的绝对色值
   本身是重建目标, 归一化抹成零均值会导致色彩漂移。
2. [推测] 发尖是极度不对称的高能量脉冲, 归一化拉回"正常"分布会导致
   夹缝闭合失败、Gibbs 振铃。

**翻转条件**: 如果有人设计出 DC-aware 的归一化 (只归一化 AC, 保留
DC 锚点), 这个限制可以放开。
```

```markdown
### [2026-04-09 10:10] 失败: 空间调色板 + grid_sample + PDE 对流全线溃败
> #training-failure #grid-sample #pde #palette #gradient-disconnect
> ⚑ 关键转折点

**现象** [已确认]: 训练后发尖重建几乎没有改善。

**死因分析** [推测]:
1. **grid_sample 在低分辨率的物理不可能性**: palette_field 生成在 1/4
   分辨率。2px 发尖在 palette_field 上只有 0.5px。双线性插值物理性地
   混合了 50%+ 的背景色。
2. **远程 offset 的梯度断裂**: 零初始化 offset 起步采样局部 → 发尖
   局部全是背景 → 梯度只引导 offset 在背景徘徊 → 永远"看不到" 50px
   外的发根颜色。
...

**认知刷新**:
- [推测] 所有依赖 grid_sample 空间插值的色彩搬运方案, 在低分辨率
  特征图上都有物理分辨率极限。
- [推测] 所有依赖局部梯度的算子在零初始化+远程目标场景下都有梯度断裂问题。
```

这里发生了三件传统文档里看不到的事:

1. **`[已确认] / [推测] / [直觉]`** — 每个关键判断都声明了证据强度。LLM 没法把推测包装成事实。
2. **`翻转条件`** — 每个假说都必须声明"什么会推翻我"。不可证伪的不是假说, 是教条。
3. **`↗ 被 ... 修正`** — 当后续条目修正了之前的, 两条都保留。原始推理永远不被删除; 修正链本身就是最高价值的记录。

## 为什么它有效 (第一性原理)

### 1. 事件溯源胜过状态修改

每一个需要从腐败状态恢复的健壮系统 — 银行账本、分布式数据库、版本控制 — 都是 append-only 的。修改历史等于销毁诊断"哪里出错"所需的证据。

绝大多数工程文档反其道而行: 用今天的理解覆盖昨天的。等今天的理解也被证明错了, 没有路径回去查证。Epistemic Trace 用 Git 处理代码的方式处理你的推理: 每个状态都被保留; 修订是显式的追加, 不是无声的覆写。

### 2. 可证伪性是假说和信念的分水岭

Karl Popper 的标准: 不可证伪的论断不是科学论断。同理适用于工程判断。*"PixelShuffle 不适合动漫超分"* 是一个信念。*"PixelShuffle 不适合动漫超分; 如果有人在它后面叠加足够强的后处理修复层 (如强力 DCN) 并证明能补偿数学限制, 则该断言被推翻"* 是一个假说。

强制每个假说声明翻转条件做了两件事:
- 它暴露了哪些信念其实是可测试的
- 它告诉未来的你 (和 LLM) 该盯着什么证据看

### 3. 人类记忆是重构性的; LLM 没有记忆

认知科学的结论很清楚: 人类的情景记忆不是录像。它是在调取时根据线索重构的, 重构本身受当前心理状态影响而失真。六个月后, 你"为什么选了架构 B"的记忆, 是被你当下信念塑形过的合理化产物。

LLM 的处境更糟 — 它根本没有跨会话的情景记忆。每个对话从一张白纸 + 你能塞进 context window 的内容开始。

正确的应对是把推理外化为一个持久、结构化的工件, 让两类读者都能消费。Epistemic Trace 从第一天起就为这两类读者设计 — 格式足够结构化让 LLM 可靠解析, 又足够散文化让人类能跳读。

### 4. 失败是密度最高的信息

在搜索问题里, 你排除掉的每一条死路都缩小了剩余的解空间。绝大多数工程项目文档只记录走通的那条路 — 这是熵最低的可能记录。

Epistemic Trace 反其道而行: 失败条目被定为最高价值的内容。一条带详细死因分析的 `失败:` 条目, 能阻止同一死路被再次提议 — 被你、你的合作者、或你的 LLM。

### 5. 不加约束的 LLM 会放大认知不诚实

LLM 默认输出自信的、确定性的散文。*"根因是 X。"* 当这段文字之后被读到 — 被人, 或被另一个 LLM 在另一个会话里 — 读者会把它当作事实。新决策会被叠加在一个被渲染成确定性的猜测上。

`[已确认] / [推测] / [直觉]` 标记系统是结构性的防御。协议要求 LLM 在每个关键判断点标注证据基础。你扫一眼 Trace 就能看出哪些承重论断其实是猜测。

## 适用场景

下列条件 *同时* 成立时, Epistemic Trace 最有用:

- 项目周期是周或月, 不是小时
- 你和 LLM agent 协同 (Copilot、Claude Code、Cursor、Aider 等)
- 决策会累积、被修正、需要被重新审视
- 一个错误推荐的代价是真实时间 (训练任务、实验、上线)

具体来说: 深度学习研究、科研软件、复杂重构、产品策略、药物发现、法律论证 — 任何"推理需要复利累积而非蒸发"的领域。

它在以下场景 **不适用**: 短任务、探索性脚本、不和 LLM 协同的项目、或者团队紧密、部落知识装得进每个人脑子里的环境。

### 与现有系统对比

| 系统 | 它记录什么 | 在 LLM 协同研究中缺什么 |
|------|------------|--------------------------|
| ADR (架构决策记录) | 决策 + 上下文 + 后果 | 无置信度标记、无 LLM 协议、无失败优先设计 |
| 实验笔记本 | 按天的探索记录 | 手写、无 LLM 可读结构、无因果存疑标记 |
| Zettelkasten / wiki | 原子化、可链接的笔记 | 历史可改 — 销毁修订证据 |
| `CHANGELOG` + git log | 改了什么、什么时候 | 记录了你做的, 不记录你试过又废弃的 |

## 快速上手

### 1. 把框架文件放进你的项目

```
your-project/
└── docs/
    └── EpistemicTrace/
        ├── SPEC.md              # 框架规范
        ├── PROTOCOL.md          # LLM 操作协议
        └── EpistemicTrace.md    # 你项目的 Trace (从空开始)
```

从本仓库复制 `SPEC.md` 和 `PROTOCOL.md`。创建一个空的 `EpistemicTrace.md`, 带上两个区的骨架。

### 2. 告诉你的 LLM agent 这件事

把 Project Brief 片段 (在 `PROTOCOL.md` 末尾) 放进你平台的自动上下文文件:

| 平台 | 文件 |
|------|------|
| GitHub Copilot | `.github/copilot-instructions.md` |
| Claude Code | `CLAUDE.md` |
| Cursor | `.cursorrules` |
| Aider / 其他 | `AGENTS.md` |

这是每个会话默认加载的唯一内容。它指示 agent 在何时、如何查阅 Trace — 让简单任务的 token 开销保持在零。

### 3. 从一次真实决策生成第一条条目

不要靠手写说明书来预填 Trace。让它从真实认知事件里生长出来。下次你做架构选择时, 让 LLM agent: *"按 PROTOCOL.md 给这次决策生成 Trace 条目。"* 审核, 合并。

### 4. 等模式浮现时生成假说

当 5-10 个条目共享同一个 `#tag` 时, 通常一个工作假说已经可见。让 LLM 起草一条 `H{N}` 条目, 总结这个约束, 包含翻转条件。审核, 合并。

# Epistemic Trace — Specification v0.1

> 一套 Append-Only 的认知日志框架。
> 记录决策背后的**推理链**, 不仅仅是结论。
> 设计为双读者: 作者本人的心流记录 + LLM Agent 的上下文源。

---

## 0. 设计哲学

**为什么记推理链而非结论**:
结论随时可以翻。推理链永远有参考价值——即使结论被推翻, 链条上的每一步分析仍然是有效的约束排除。对 LLM 而言, 读到推理链意味着它能判断"当前假说是否也可能被同类证据推翻", 而非只能接受一个无根的断言。

**Append-Only 的理由**:
修改历史等于销毁证据。事后看来"错误"的推理, 恰恰是最高价值的信息——它标记了认知地图上的死路, 防止未来重蹈。

---

## 1. 核心规则

| # | 规则 | 理由 |
|---|------|------|
| R1 | **只追加, 不修改**历史条目的主体内容 | 事件溯源完整性 |
| R2 | 唯一允许的回溯操作: 在旧条目末尾加一行 `↗ 被 [日期] 条目修正` | 保留因果链, 读者可跳转 |
| R3 | **后文认知自动覆盖前文假设**, 以最新条目为准 | 避免维护"哪个版本是最新"的元数据 |
| R4 | `#tag` 做高维寻址, 跨模块、跨时间关联 | 扁平标签 > 树状目录, 支持多维交叉 |

---

## 2. 文档结构

一份 Epistemic Trace 文档由两个区组成:

### 2.1 工作假说区 (Hypotheses)

位于文档顶部。存放**当前有效**的有条件推断。

每条假说格式:

```markdown
### H{N}: {一句话命题}
> #tag1 #tag2

{论证主体: 为什么这样认为, 证据是什么}

**翻转条件**: {在什么实验或证据下, 这条假说应被推翻}
```

**必要字段**:
- **编号** `H{N}`: 全局唯一, 只增不减。被废弃的假说不删号, 在正文标注 `[已废弃]` + 指向替代条目
- **命题**: 一句话, 可以是断言也可以是约束 ("避免X", "必须Y")
- **标签**: 至少一个 `#tag`
- **翻转条件**: **必填**。没有翻转条件的假说不是假说, 是教条

**可选字段**:
- **重要补充 [日期]**: 对假说做非颠覆性的增量修正。追加在原文之后, 不修改原文
- **表格/列表**: 辅助说明

**假说更新协议**:
- 增量修正 → 在原假说末尾追加 `**重要补充 [YYYY-MM-DD]**:` 段落
- 颠覆性修正 → 在时间线区写新条目, 旧假说加 `[已废弃] ↗ 被 [日期] 条目替代`, 新假说拿新编号

### 2.2 时间线区 (Timeline)

按时间倒序排列 (最新在最下)。每个条目是一次**认知事件**。

## 3. 时间线条目格式

```markdown
### [YYYY-MM-DD HH:MM] {标题: 动作 + 对象}
> #tag1 #tag2 #tag3
> ↗ 被 [YYYY-MM-DD] 条目修正        ← 可选, 仅当后续条目推翻本条时追加

{正文}

**认知刷新**:
{这次事件后, 心智模型更新了什么。1-3 句话。}
```

### 3.1 条目类型 (非强制分类, 仅供参考)

| 类型 | 典型标题前缀 | 说明 |
|------|-------------|------|
| 设计决策 | `架构:`, `实现:`, `选型:` | 选择了什么, 为什么 |
| 实验失败 | `失败:` | **高价值**。记录死路和死因 |
| 实验结果 | `结果:` | 数据/观测 + 解读 |
| 认知转折 | (无前缀, 用 `⚑ 关键转折点` 标注) | 推翻了之前的核心假设 |
| 废弃记录 | `中间方案 (已废弃):` | 未经验证即废弃的方案, 附废弃原因 |

### 3.2 正文结构 (推荐但不强制)

一个完整的决策条目通常包含:

1. **起因**: 触发这次思考的现象或问题
2. **分析/诊断**: 推理过程, 尽量写明因果链
3. **决策**: 做了什么, 或选择了哪个方案
4. **效果** (如有): 实验结果或自检结论
5. **认知刷新**: 心智模型的增量更新

失败条目额外包含:
- **现象**: 可观测的异常表现
- **根因分析**: 编号列出每个致命缺陷
- **认知刷新**: 从失败中提取的通用教训

### 3.2.1 置信度标记 (认知诚实)

分析和结论必须区分**已确认的事实**与**推测性判断**。用以下标记:

| 标记 | 含义 | 使用场景 |
|------|------|----------|
| `[已确认]` | 有直接证据支撑 | 可复现的实验结果、可观测的数据 |
| `[推测]` | 基于证据的合理推断, 但未直接验证 | 根因分析、因果归因 |
| `[直觉]` | 无硬证据, 基于经验或类比 | 早期假说、方向性判断 |

**为什么需要这个**:
LLM 生成文本时天然倾向于使用确定性语言 ("根因是 X"、"这证明了 Y"), 即使实际只是推测。当后续读者 (人或 LLM) 读到以确定性语言写成的推测时, 会将其视为已确认的事实, 导致:
- 在错误的前提上叠加新决策
- 忽略替代解释
- 过早关闭探索空间

置信度标记不需要每句话都加。只在**关键判断点** (根因归因、方案选择理由、认知刷新) 标记即可。

### 3.3 关键转折标注

当一个条目推翻了之前的核心假设或改变了项目方向, 在标签行下方加:

```markdown
> ⚑ 关键转折点
```

读者 (人或 LLM) 可以只扫 `⚑` 标记来快速定位重大认知转变。

### 3.4 因果存疑标注

当一个问题最终被其他方式解决 (如扩充数据集、延长训练), 导致先前为解决该问题所做的工作无法确认是否有效时, 在相关条目的标签行追加:

```markdown
> ◇ 因果存疑: 问题已由 {实际解法} 解决, 本条目的贡献不确定
```

**语义**: 不是推翻 (`↗`), 不是转折 (`⚑`), 不附带行动项。纯粹的认知诚实标记——"这个工作被保留在最终架构中, 但我不知道它到底有没有帮上忙"。

**对 LLM 的含义**: 标注 `◇` 的条目中的设计决策仍然是当前架构的一部分, 但不应作为"已验证有效"的证据来引用。

## 4. 标签系统

### 4.1 规范

- 格式: `#kebab-case`, 全小写, 连字符分隔
- 位置: 条目标题下方的 blockquote 行, 空格分隔
- 一个条目可以有任意数量的标签
- 标签不需要预定义——按需创建, 自然涌现

### 4.2 标签用途

- **跨时间关联**: 同一个 `#tag` 串起了该主题从初始假设到被推翻的完整链条
- **跨模块关联**: `#loss #topology` 同时出现, 意味着这是 loss 和拓扑的交叉认知
- **LLM 检索锚点**: LLM 可以被指示"只关注 #tag 相关的条目"

### 4.3 标签不做的事

- 不是分类系统 (一个条目不需要"属于"某个类别)
- 不是层级结构 (没有 `#parent/child`)
- 不做版本标记 (用时间线本身的顺序)

---

## 5. 回溯标注协议

当新条目推翻或修正了旧条目:

**在旧条目**的标签行末尾追加:
```markdown
> ↗ 被 [YYYY-MM-DD HH:MM] 条目修正
```

**在新条目**中显式引用旧条目:
> "针对 [YYYY-MM-DD] 的三个缺陷做彻底重设计"

这形成双向链接: 旧→新 (回溯标注) + 新→旧 (正文引用)。

## 6. LLM 协同设计

Epistemic Trace 是 **LLM 生成, 人 review** 的文档。LLM 既是主要作者, 也是主要消费者。人的职责是触发写入 (通过对话/任务/事件) 和审核内容, 大部分不需要手写正文。

### 6.1 平台无关的分层架构

不同 LLM Agent 平台 (VS Code Copilot, Claude Code, Cursor, 等) 都有两种上下文注入方式:

| 通用概念 | 作用 | 各平台实现 |
|---------|------|------------|
| **自动上下文** — 每次会话默认加载 | 项目身份 + Trace 入口指引 | AGENTS.md / CLAUDE.md / .cursorrules / 等 |
| **按需上下文** — Agent 主动读取文件 | 操作协议 + Trace 内容 | 任何平台的 read_file 能力 |

文件结构:

```
docs/EpistemicTrace/
├── SPEC.md              # 框架规范 (本文件, 给人和 LLM 看)
├── PROTOCOL.md          # LLM 操作协议 (从 SPEC 提炼, 纯执行规则)
├── EpistemicTrace.md    # L0 完整 Trace
└── L1_Projection.md     # L1 压缩视图
```

**设计原则**: Trace 相关 token 在简单任务中开销为零。只有涉及设计决策、问题诊断等需要项目认知上下文的任务时, Agent 才读取 PROTOCOL.md 和 Trace 文件。

### 6.2 Project Brief (放入各平台自动上下文文件)

以下标准文本应放入各平台的自动上下文文件中:

```markdown
## Epistemic Trace

本项目使用 Epistemic Trace 认知日志框架记录所有设计决策和推理链。

- 路径: `docs/EpistemicTrace/`
- 涉及设计决策、方案选型、问题诊断、技术探索等需要项目认知上下文的任务时, 先读 `docs/EpistemicTrace/PROTOCOL.md`, 再读 L0 或 L1
- 工作完成后如产生认知事件, 按 PROTOCOL.md 生成条目提交 review
- 不要推荐 Trace 中已记录为失败的方案
```

**Project Brief 不放**: Trace 内容、假说、时间线、读写协议细节 — 这些都在 Agent 按需读取的文件中。

### 6.3 LLM 操作协议

完整的读写规则见 `PROTOCOL.md`。该文件从本 SPEC 的 §2-§5 提炼而来, 不含设计哲学, 只有 Agent 需要执行的操作规则。

Agent 在新会话中的工作流:

```
新会话 → 自动加载 Project Brief
  → 用户提问
    → 简单任务 → 直接工作, 不读 Trace
    → 需要项目认知上下文的任务 → 读 PROTOCOL.md → 读 L0 或 L1 → 工作 → 生成条目 → 人 review
```

## 7. 已知局限与未来方向

> 此节记录框架本身的待解决问题, 不涉及具体项目。

### 7.1 Token 预算: 双层 Crystallization 协议

**矛盾**: 推理链的价值在于细节, 但 LLM context window 有限。

**解法: L0 Full + L1 Projection**

| 层 | 内容 | 生成者 | 触发时机 | 压缩 |
|---|------|--------|---------|------|
| **L0 Full** | 完整推理链 | LLM 生成, 人 review | 实时 (对话/任务/事件发生时) | **永不压缩** |
| **L1 Projection** | 压缩视图 | LLM 从 L0 派生, 人 review | L0 超过 context 预算时 | 有损, 可从 L0 重建 |

**核心不变量**: L1 是 L0 的纯函数。任何时候都可以从 L0 重新生成 L1 (只花 token, 不丢信息)。

#### Crystallization 规则 (L0 → L1)

| 内容类型 | 压缩策略 |
|---------|----------|
| 假说区 | **原样保留** (已是高密度, 不可压缩) |
| 活跃工作区 (最近 N 天的条目) | **原样保留** (当前工作需要完整上下文) |
| 已稳定的条目链 (尝试→失败→修复→成功) | LLM 压缩为摘要段落, 保留标签、回溯链、认知刷新, 压缩分析正文 |
| 标签索引 | 自动生成, 放在 L1 顶部 (`#tag: [日期1], [日期2], ...`) |

每条压缩摘要附元数据:
```markdown
> [crystallized: YYYY-MM-DD, source: L0 条目 YYYY-MM-DD~YYYY-MM-DD]
```

#### 活跃窗口 (N 值)

N 的选取原则: 当前工作可能回溯的最大时间跨度。建议起步 N=14 天, 根据实际使用调整。活跃窗口内的条目从 L0 直接复制到 L1, 不做任何压缩。

#### 使用策略

| 场景 | 喂什么 |
|------|--------|
| L0 未超 context | 直接喂 L0 (当前阶段) |
| L0 超了 | 喂 L1 全文 |
| 需要深挖某 #tag | 喂 L1 + 从 L0 按 tag 检索相关条目全文 |

#### 语义漂移防护

LLM 摘要是有损压缩, 存在语义漂移风险 (LLM 按当前重要性丢弃信息, 但未来认知转折可能需要那些细节)。防护措施:
1. L0 永不压缩 — 漂移被隔离在 L1, 随时可从 L0 重建
2. 每条摘要带 `[crystallized]` 时间戳和来源范围 — 读者知道这是哪个时间点的压缩视图
3. 当 L1 摘要与当前认知产生矛盾时, **必须回溯 L0 原文**, 不可基于 L1 做判断

### 7.2 跨项目知识迁移

部分假说是领域知识 (如"动漫图像是分段平滑流形"), 不是项目专属的。

**待设计**: 领域知识层与项目知识层的分离机制。可能的形式: 领域级 Epistemic Trace (如 `anime-sr.epistemic.md`) + 项目级 Trace。

### 7.3 多 Agent 协同

当多个 LLM Agent 同时参与开发时, 需要:
- 写入冲突解决 (谁有权追加条目)
- 不同 Agent 的认知隔离与共享边界
- 跨平台一致性保证 (不同平台的 Agent 读写同一份 Trace)

### 7.4 索引与检索

当文档膨胀到数万字时, 全文线性阅读效率极低。

L1 Projection 的标签索引 (§7.1) 部分解决了此问题: LLM 可按 `#tag` 定位相关条目, 再从 L0 钻取全文。

**待设计**: 更复杂的检索场景 (如跨项目搜索、语义检索) 留待实际遇到瓶颈时再解决。

# Epistemic Trace — LLM Protocol

> Agent 操作协议。不含设计哲学, 只有执行规则。
> 完整规范见 `SPEC.md`。

---

## 何时读 Trace

涉及以下任务时, **先读 Trace 再动手**:
- 设计决策、方案选型、架构变更
- 问题诊断、失败分析、性能调试
- 技术探索、可行性评估
- 任何涉及"为什么之前这样做"的问题

简单任务 (格式修复、依赖更新、工具命令) 不需要读 Trace。

## 怎么读

### 文档结构

Trace 有两个区:
1. **工作假说区** (顶部): 当前有效的有条件推断, 每条带翻转条件
2. **时间线区** (下方): 按时间顺序的认知事件, 后文自动覆盖前文

### 标注符号

| 标记 | 含义 | 你应该怎么做 |
|------|------|-------------|
| `↗ 被 [日期] 条目修正` | 已被后续推翻 | 以后续条目为准 |
| `⚑ 关键转折点` | 重大认知转变 | 优先阅读 |
| `◇ 因果存疑` | 保留在架构中, 但不确定是否有效 | 不作为"已验证有效"的证据引用 |
| `**翻转条件**` | 假说的可证伪边界 | 检查是否已被当前证据触发 |
| `**认知刷新**` | 心智模型增量更新 | 快速扫描可建立当前世界观 |

### 关键规则

1. **不要推荐已记录为失败的方案**, 除非你有新证据表明翻转条件已触发
2. 假说是有条件推断, 不是定律 — 注意翻转条件
3. 后文自动覆盖前文, 以最新条目为准

## 怎么写

当对话中产生认知事件时, 生成 Trace 条目提交人 review。

### 触发条件

以下情况应生成条目:
- 做出或变更了设计决策
- 实验/测试/验证产生了有意义的结果 (成功或失败)
- 诊断出了问题的根因
- 推翻了之前的假设或认知

### 条目格式

```markdown
### [YYYY-MM-DD HH:MM] {标题: 动作 + 对象}
> #tag1 #tag2

{正文: 起因 → 分析 → 决策 → 效果}

**认知刷新**:
{心智模型更新了什么, 1-3 句}
```

### 写入规则

1. **失败记录是最高价值内容** — 详细记录死因和根因分析
2. 推翻旧条目时, 在旧条目标签行追加 `↗ 被 [YYYY-MM-DD HH:MM] 条目修正`
3. 触发假说翻转条件时, 记录事件并处理假说 (增量修正或废弃)
4. 重大转折加 `⚑ 关键转折点`
5. 因果不明加 `◇ 因果存疑: 问题已由 {实际解法} 解决, 本条目的贡献不确定`
6. **生成后提交人 review, 不要自行写入文件**

### 认知诚实

生成条目时必须遵守:

1. **区分事实与推测**: 在关键判断点 (根因归因、方案选择理由、认知刷新) 标注 `[已确认]` / `[推测]` / `[直觉]`
2. **不要用确定性语言包装推测**: "根因可能是 X" 而不是 "根因是 X" (除非有直接证据)
3. **显式列出替代解释**: 当根因分析基于推测时, 至少提及一个替代假说
4. **不要过早关闭探索空间**: “这肯定是 X 的问题” → “目前证据指向 X, 但也可能是 Y”
5. **认知刷新中不要过度泛化**: 从一次实验得出的结论, 不要写成放之四海皆准的真理

## Project Brief 模板

以下文本应放入各平台的自动上下文文件中 (AGENTS.md / CLAUDE.md / .cursorrules 等):

```markdown
## Epistemic Trace

本项目使用 Epistemic Trace 认知日志框架记录所有设计决策和推理链。

- 路径: `docs/EpistemicTrace/`
- 涉及设计决策、方案选型、问题诊断、技术探索等需要项目认知上下文的任务时, 先读 `docs/EpistemicTrace/PROTOCOL.md`, 再读 L0 或 L1
- 工作完成后如产生认知事件, 按 PROTOCOL.md 生成条目提交 review
- 不要推荐 Trace 中已记录为失败的方案
```
